# Workplan: meteo readings linked to radiometer measurements

## Goal and data model

Each radiometer measurement may reference the latest valid WN90LP reading. The
relationship is intentionally many-to-one:

```text
devices 1 ── N measurements
devices 1 ── N meteo_readings
meteo_readings 1 ── N measurements
```

- `measurements.meteo_reading_id` is nullable and points to `meteo_readings.id`.
- One WN90LP sample is reused by all radiometer measurements taken before the next
  station poll; `(device_id, timestamp_ms)` deduplicates that sample.
- `devices.id` is not generated by PostgreSQL. It comes from the MQTT topic prefix
  (`<device_id>/measure`), which the firmware builds from `app_config.device_id`.
  `touch_device()` creates the device row on first telemetry.
- `measurements.id` and `meteo_readings.id` are UUIDs generated by the backend.

## Review findings and status

### Critical findings fixed in the corrective slice

- [x] **MQTT poison-message could terminate the worker.** Non-object/invalid UTF-8,
  non-finite numbers and out-of-range meteo timestamps are rejected; each MQTT
  message is isolated so one bad retained payload cannot create a restart loop.
- [x] **Existing devices would keep the legacy 60-second poll.** A saved config with
  `meteo_poll_interval_s=60` and no `meteo_log_interval_s` is recognized as the old
  format and migrated to the new 9-second default.
- [x] **The nominal 60-second CSV cadence was actually 63+ seconds and used wall
  clock arithmetic.** Poll and log deadlines now use independent monotonic schedules;
  NTP/GPS corrections no longer distort the cadence.

### Additional findings fixed in the same slice

- [x] Failed storage writes no longer advance the log deadline or emit a false
  `logged` message; retry occurs after the next poll.
- [x] `NaN` and `Infinity` meteo fields are converted to `NULL` instead of reaching
  PostgreSQL; invalid integer values no longer raise.
- [x] A repeated `(device_id, timestamp_ms)` delivery refreshes all meteo fields,
  allowing a correct retry to repair an incomplete first delivery.
- [x] The new cJSON meteo subtree checks allocation/add failures and frees an
  unattached subtree.
- [x] Meteo-only config keys count as successfully parsed configuration.

### Open risks before production rollout

- [ ] **P1 — MQTT authorization.** The browser receives shared public MQTT credentials
  and Mosquitto has no topic ACL. Split browser/device/worker identities and restrict
  publish/subscribe topics; parser hardening prevents the crash but does not prevent
  forged telemetry or arbitrary device creation.
- [ ] **P2 — online migration locking.** Migration `00021` creates a regular index and
  validates the FK while holding DDL locks on `measurements`. Test against a production-
  sized copy and, if needed, use a staged/concurrent index plus deferred FK validation.
- [ ] **P2 — CSV sentinel semantics.** Existing CSV code writes invalid WN90LP floats
  as `0.0`; change the format to an empty/explicit invalid value so real zero remains
  distinguishable from missing data.
- [ ] **P3 — same-device database invariant.** The current worker always supplies the
  same topic-derived device ID to the measurement and meteo reading, but PostgreSQL
  does not enforce that invariant. Decide whether to add a composite FK/unique key.
- [ ] **P3 — device ID validation.** Bound and validate the MQTT topic prefix before
  `touch_device()` so authenticated publishers cannot create arbitrary or oversized IDs.

## Active rollout checklist

- [ ] Add role-separated Mosquitto users and topic ACLs; rotate the currently shared
  credential during deployment.
- [ ] Add a PostgreSQL integration test for meteo upsert, measurement FK linking and
  duplicate-delivery repair.
- [ ] Exercise migration `00020 -> 00021 -> 00020` on a production-sized PostgreSQL
  snapshot while telemetry writes are running.
- [ ] Hardware smoke-test an upgraded device with a legacy saved config:
  - UART reports the `60s -> 9s` migration once;
  - WN90LP polls near 9 seconds;
  - CSV writes stay near 60 seconds;
  - temporary storage contention retries without losing the full minute;
  - `/measure` contains a finite, valid `meteo` snapshot only while online.
- [ ] Verify old firmware without a `meteo` object still stores measurements with
  `meteo_reading_id = NULL`.
- [ ] Ensure the untracked `00021_meteo_readings.py` migration is included in the
  eventual commit/deployment artifact.
- [ ] Optional second phase: expose linked meteo data through the API and frontend.
